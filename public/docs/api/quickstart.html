Authentication

University Assist API uses Supabase authentication with API keys and JWT tokens.

Headers Required

apikey: {SUPABASE_ANON_KEY} Authorization: Bearer {JWT_TOKEN} Content-Type: application/json
Note: Replace {SUPABASE_ANON_KEY} with your actual Supabase anonymous key and {JWT_TOKEN} with a valid JWT token from authentication.

cURL Example

curl -H "apikey: {SUPABASE_ANON_KEY}" \ -H "Authorization: Bearer {JWT_TOKEN}" \ -H "Content-Type: application/json" \ "https://zfiexgjcuojodmnsinsz.supabase.co/rest/v1/cities"

JavaScript/Node.js Example

const response = await fetch('https://zfiexgjcuojodmnsinsz.supabase.co/rest/v1/cities', { method: 'GET', headers: { 'apikey': '{SUPABASE_ANON_KEY}', 'Authorization': `Bearer ${jwtToken}`, 'Content-Type': 'application/json' } }); const cities = await response.json();

Python Example

import requests headers = { 'apikey': '{SUPABASE_ANON_KEY}', 'Authorization': f'Bearer {jwt_token}', 'Content-Type': 'application/json' } response = requests.get( 'https://zfiexgjcuojodmnsinsz.supabase.co/rest/v1/cities', headers=headers ) cities = response.json()

PostgREST Query Parameters

The REST API supports powerful querying via URL parameters:

  • select - Choose columns: select=id,name,slug
  • order - Sort results: order=name.asc,created_at.desc
  • limit - Limit results: limit=10
  • offset - Pagination: offset=20
  • eq - Equal: city_id=eq.{uuid}
  • neq - Not equal: published=neq.false
  • gt/gte - Greater than: tuition_fees=gte.1000
  • lt/lte - Less than: ects_credits=lte.240
  • like/ilike - Pattern match: name=ilike.*berlin*
  • in - In list: degree_level=in.(bachelor,master)
  • is - Is null: logo_url=is.null
  • or - Logical OR: or=(field1.eq.value1,field2.eq.value2)

Rate Limits & Best Practices

Supabase Rate Limits: Respect the usage limits of your Supabase plan. For production use, consider upgrading to a paid plan for higher limits.

Best Practices

  • Use select to fetch only needed columns
  • Implement pagination with limit and offset
  • Cache responses when appropriate
  • Use proper HTTP status code handling
  • Implement exponential backoff for retries

Row Level Security (RLS)

Important: All database operations are subject to Row Level Security policies. Operations will only succeed if the authenticated user has the appropriate permissions according to the RLS policies.

Key RLS behaviors:

  • Public data: Cities, universities, and published programs are publicly readable
  • User data: Profiles, watchlists, and applications are user-specific
  • Admin data: Administrative functions require admin role
  • Ambassador data: Only published ambassadors are publicly visible

Error Handling

The API returns standard HTTP status codes and error messages:

{ "error": "permission_denied", "message": "Insufficient privileges", "details": "Row level security policy violation", "hint": "Ensure you have the required permissions", "code": "42501" }
